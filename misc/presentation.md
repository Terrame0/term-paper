## Постановка задачи

Демонстрация реального use-case БД, не только её схемы:
- Запроектирована **предметная область** (рекламное агентство)
- Реализован **нагрузочный тестер** — параллельные клиенты бьют по prod-БД
- Результаты прогонов сохраняются в **отдельной БД**
- Визуализация — серверный веб-интерфейс с heatmap'ами

**Цель:** увидеть как реальные параметры (число клиентов, объём данных, длина сессии) влияют на latency запросов.

| Слой | Технология |

## Предметная область: рекламное агентство

12 таблиц, три уровня:

```
client
  ├─ phone, email, address          (контакты)
  └─ promotion                      (рекламные акции)
        ├─ promotion_billboard
        ├─ promotion_newspaper
        ├─ promotion_radio
        ├─ promotion_youtube
        ├─ promotion_social_media
        ├─ promotion_email_campaign
        └─ promotion_result
```

Один клиент → много акций → каждая акция конкретного типа (биллборд / газета / радио / YouTube / соцсеть / email-кампания) + измеренные результаты (показы / клики / конверсии).

## Архитектура тестера

CLI: `db-tester --workers W --ops-per-worker O --prefill P --scenario basic`

Каждый параметр — либо `N` (фиксированное), либо `base-mult-count` (геометрическая прогрессия: `10-3-4` → `[10, 30, 90, 270]`).

**Один прогон = один `test`.**
В зависимости от числа диапазонов:
- **2 диапазона** → один `test_sweep` (одна 2D-сетка)
- **3 диапазона** → три `test_sweep` (все попарные сечения)

---

## Модель параллелизма

- `N` корутин-воркеров на `asyncio`
- Каждый воркер **открывает своё соединение** через `asyncpg.connect(..., command_timeout=5)`
- Держит весь прогон → закрывает в конце
- Это моделирует **долгоживущего клиента** (сессия в админке / приложении)

**Альтернатива** — пул соединений — даёт похожие цифры, но менее показательна для защиты.

**Контекст:** asyncio + asyncpg обходят Python GIL (для I/O-bound задач это идиоматично) и не платят за оверхед потоков ОС.

---

## Модель данных результатов

```
test (1) ── test_sweep (1-3) ── test_cell (N × M)
```

- `test` — один запуск CLI
- `test_sweep` — одна 2D-сетка (`x_axis`, `y_axis`, `fixed_param`)
- `test_cell` — одна точка сетки с измеренными `p50_ms`, `p95_ms`

Каждая ячейка = новая «чистая» БД (TRUNCATE + prefill) → `N` воркеров параллельно делают по `M` операций → агрегация в перцентили.

## Сценарий нагрузки

В `python/src/scenarios.py`, словарь `SCENARIOS["basic"]` — три SQL с весами:

```python
"basic": [
    Op("insert_client", 1, "INSERT INTO client ..."),
    Op("select_random_client", 3, "SELECT * FROM client ORDER BY RANDOM() LIMIT 1"),
    Op("join_client_promotion", 2, "SELECT ... JOIN promotion ..."),
]
```

- Веса 1:3:2 — read-heavy с примесью записи и join'а
- Все запросы **самодостаточны** (RANDOM на стороне SQL) — нет зависимости между операциями воркера

## Эксперимент: test #3

3 sweep'а × ~30 ячеек = **81 ячейка**, 10 минут прогона.

| Sweep | Оси | Fixed |
|---|---|---|
| #5 | `workers × ops_per_worker` | `prefill = 1000` |
| #6 | `workers × prefill` | `ops_per_worker = 3125` |
| #7 | `ops_per_worker × prefill` | `workers = 64` |

`workers ∈ [1, 64]` (×2), `ops_per_worker ∈ [1, 3125]` (×5), `prefill ∈ [10, 1000]` (×10).

---

## Sweep #5: workers × ops_per_worker

> **Скриншот heatmap #5 сюда**

**Наблюдения:**
- При `workers ≤ 4`: рост `ops_per_worker` **снижает** latency
  → прогрев соединения, prepared-cache
- При `workers ≥ 32`: рост `ops_per_worker` **повышает** latency
  → lock contention, давление на shared_buffers
- Точка перегиба — около 8 воркеров

---

## Sweep #6: workers × prefill

> **Скриншот heatmap #6 сюда**

**Наблюдения:**
- `prefill 10 → 1000` (×100) даёт почти линейный, но **слабый** эффект (+10-15% latency)
- `workers 1 → 64` (×64) даёт ×55 в latency
- **Workers доминирует над prefill**

---

## Sweep #7: ops_per_worker × prefill (workers=64)

> **Скриншот heatmap #7 сюда**

**Наблюдения:**
- При `workers=64` (точка деградации) prefill начинает «кусать»
- Главный «красный угол»: `ops=3125, prefill=1000` → p95 = 60 ms
- В одиночку ни `ops`, ни `prefill` так не разгоняются — это **взаимодействие** на высокой конкуренции

---

## Главный вывод

> Производительность БД определяется в основном **числом параллельных клиентов**. Объём данных и длина сессии дают эффект **только при высоких** значениях `workers` — это нелинейная зависимость, не аддитивная.

Точка перегиба — между 32 и 64 воркерами. Характерно для PostgreSQL с дефолтным `max_connections=100` и process-per-connection моделью: каждый коннект = форк бэкенд-процесса, конкуренция за `shared_buffers` и WAL-лок.

## Технические детали окружения

**Nix flake** обеспечивает:
- Один `flake.nix` — весь стек поднимается
- Никаких внешних зависимостей кроме nix
- Воспроизводимая сборка PHP / Python / pgschema на любой машине

**pgschema** — декларативные миграции:
- В корне лежат `production-db-ddl.sql` и `test-db-ddl.sql` (целевое состояние)
- При `server-start` pgschema diff'ит реальную БД с DDL и накатывает только разницу
- Откат / пересоздание — снос `/tmp/term-paper` и заново

---

## Состояние и эфемерность

Всё runtime-состояние в **`/tmp/term-paper/`**:
- `postgres/data/` — кластер
- `postgres/sockets/` — Unix-сокет
- `nginx/`, `php/` — pid-файлы, логи, fpm-сокет

`/tmp` эфемерно → **полный сброс одной командой**:
```bash
server-stop
rm -rf /tmp/term-paper
server-start    # initdb + schema-apply отработают заново
```

Удобно для повторных экспериментов: каждый прогон — гарантированно чистая БД.

## Заключение

Реализован **полный замкнутый цикл**:
1. БД с реальной предметной областью
2. Нагрузочный тестер с конкуренцией
3. Сохранение и визуализация результатов в той же СУБД
4. Воспроизводимое окружение через Nix

Защищаемая идея: **проектируем БД не только под схему, но и под характер нагрузки**. Heatmap'ы дают наглядное представление о том, где находится «потолок» системы.
